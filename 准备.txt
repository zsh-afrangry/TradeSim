0、开发背景
    需要设计一个回测系统，用于模拟股票交易，判断在 用户设定的交易配置 与 交易策略下，能获取怎样的收益率
    该配置与收益率可作为未来优化的指导
1、开发环境
    python + polars，采用fast api +vue3
2、可能涉及的算法
    初步选取网格交易 为唯一算法，但是需要为 未来保留其它算法的接口
    算法的选择 需要作为input 输入
3、数据源的选择
    首选数据源为AkShare，需要pip install akshare
    代码示例:
        import akshare as ak
        # 获取 000400 的历史日线数据（qfq = 前复权）
        df = ak.stock_zh_a_hist(symbol="000400", period="daily", start_date="20240101", end_date="20260220", adjust="qfq")
4、建模
    输入：
        1、基础环境配置
            start_date:String (YYYY-MM-DD),模拟开始日期
            end_date:String (YYYY-MM-DD),模拟结束日期
            initial_capital:Float,初始投入本金（如 100000.00）
            commission_rate:Float,交易佣金费率（如 0.00025，即万分之二点五），用于计算摩擦成本
            slippage:Float,滑点设置（如 0.01），模拟真实交易中的价格偏差
        2、标的数据源
            symbol:String,"股票代码（如 ""600585""、""000400""），交由 AkShare 获取日线或分钟级行情
            data_frequency:Enum / String,"数据频度（如 ""daily"", ""5min"", ""1min""），决定 Polars 处理数据的时间粒度
        3、算法与策略引擎
            strategy_name:Enum / String,"策略路由标识（如 ""GRID_TRADING"", ""MACD_TREND""）。系统根据此标识调用对应的算法类
            strategy_params:Dictionary / JSON,核心扩展点。一个键值对字典，透传给具体算法
                strategy_params 针对网格交易的特定输入示例：
                {
                    "lower_bound": 20.00,
                    "upper_bound": 30.00,
                    "grid_step_pct": 0.05,
                    "base_position_ratio": 0.5,
                    "funds_per_grid": 10000.00
                    网格交易 (GRID_TRADING)：需要 base_position_ratio（底仓比例，如 0.5）。
                    海龟交易法则 / 均线突破 (TREND_FOLLOWING)：可能不需要静态底仓概念，而是通过动态计算 ATR（真实波动幅度）来决定每次开仓的资金比例
                }
    输出：
        1. 核心绩效指标 (Metrics)
            total_return,Float,模拟期内总收益率。
            annualized_return,Float,年化收益率。
            max_drawdown,Float,最大回撤率，衡量策略风险。
            win_rate,Float,胜率（盈利交易次数 / 总交易次数）。
            total_trades,Integer,触发交易的总次数。
        2. 资金曲线 (Time Series)
            equity_curve,Array of Objects,"每日/每时刻的账户净值快照。格式如 [{""date"": ""2025-01-01"", ""net_value"": 100000, ""drawdown"": 0}, ...]，直接喂给前端 ECharts 渲染折线图。"
        3. 交易流水 (Trade Log)
            execution_records,Array of Objects,详细的成交明细。
    基于上述 I/O，系统的内部流转将是这样一个解耦的过程：
        1、Vue3 发起请求： 将上述 Input JSON 通过 Axios POST 到 FastAPI。
        2、数据准备 (Data Fetching)： FastAPI 接收参数，解析 symbol 和时间范围，调用 AkShare 获取行情。
        3、数据转换 (Polars 介入)： 将 AkShare 返回的 Pandas DataFrame 或原始数据迅速转换为 Polars DataFrame，利用 Polars 的多线程和向量化能力进行数据清洗和技术指标计算。
        4、策略路由与撮合 (Routing & Execution)： 根据 strategy_name 实例化对应的策略类，将 strategy_params 传入。主循环遍历时间序列（或进行向量化回测），生成交易信号并更新账户资金。
        5、结果返回 (Serialization)： 整合 Metrics、Equity Curve 和 Trade Log，序列化为 JSON 返回给前端渲染。
5、数据库设计
    最契合当前架构（FastAPI + Python）且利于后续扩展的设计方案是：**以关系型数据库（如 PostgreSQL 或 MySQL）为主，结合 JSON/JSONB 字段的“半结构化（混合）”设计。** 这样既保留了关系型数据库强大的聚合查询和关系映射能力，又赋予了策略参数和交易流水极大的灵活性。
### 一、 为什么要采用“关系型 + JSON”的混合架构？

1. **结构化部分（用于检索和排序）：** 像 `symbol`（股票代码）、`total_return`（总收益率）、`max_drawdown`（最大回撤）这些字段是标准化的。用户未来肯定会需要类似“按收益率倒序查看我的收藏”或“筛选海螺水泥的模拟记录”这样的功能，关系型字段配合索引（Index）能提供极高的查询性能。
2. **非结构化部分（用于动态扩展）：** * `strategy_params`（策略参数）：网格交易有上下限和步长，未来如果加入双均线策略，参数就变成了长短周期。用 JSON 存储可以避免每次新增策略都要去改表结构（DDL操作）。
* `execution_records`（交易流水）和 `equity_curve`（资金曲线）：这些数据通常只在用户点击“查看详情”并渲染前端图表时才会整体读取，不需要在数据库层面对单笔交易进行复杂的 SQL 过滤。将它们打包成 JSON 存储，存取效率极高。



### 二、 核心数据库表结构设计

建议将数据拆分为主从两张表，以避免海量的交易流水拖慢主表的查询速度（特别是列表页查询）。

#### 表 1：`simulation_records` (模拟记录主表 - 核心业务表)

这张表用于前端展示“收藏列表”，存储所有需要结构化检索的元数据和绩效指标。

| 字段名 | 数据类型 (SQL) | 描述 |
| --- | --- | --- |
| `id` | BIGINT (PK) | 记录唯一标识，自增或 Snowflake ID |
| `user_id` | BIGINT (FK) | 关联用户表，标识谁收藏的 |
| `strategy_name` | VARCHAR(50) | 策略标识（如 "GRID_TRADING"） |
| `symbol` | VARCHAR(20) | 交易标的（如 "600585"） |
| `start_date` | DATE | 回测起始日期 |
| `end_date` | DATE | 回测结束日期 |
| `strategy_params` | **JSON / JSONB** | 动态策略配置参数字典 |
| `total_return` | DECIMAL(10,4) | 总收益率（建立索引，方便倒序排行） |
| `annualized_return` | DECIMAL(10,4) | 年化收益率 |
| `max_drawdown` | DECIMAL(10,4) | 最大回撤率 |
| `created_at` | TIMESTAMP | 收藏时间 |

#### 表 2：`simulation_logs` (模拟详情从表 - 存储大体积 Payload)

这张表专门用来存储大体积的数组数据。只有当用户点击某一条具体的收藏记录，进入“详情页”需要画图时，才通过 `record_id` 联表或单独查出。

| 字段名 | 数据类型 (SQL) | 描述 |
| --- | --- | --- |
| `record_id` | BIGINT (PK/FK) | 对应 `simulation_records.id`（1对1关系） |
| `equity_curve` | **JSON / JSONB** | 资金曲线数组，格式：`[{"date": "...", "net_value": ...}]` |
| `execution_records` | **JSON / JSONB** | 成交流水数组，格式：`[{"timestamp": "...", "action": "BUY", ...}]` |

### 三、 这种设计的优势与实际工作流

1. **写入（保存收藏）：** FastAPI 接收到前端 Vue3 传来的完整 JSON 结果后，将其拆分。标准字段映射到主表模型，`params`、`curve`、`logs` 保持 JSON 格式，利用 SQLAlchemy 的跨表事务一次性 `COMMIT` 插入这两张表。
2. **读取（列表页）：** 执行类似 `SELECT id, symbol, total_return FROM simulation_records WHERE user_id = ? ORDER BY total_return DESC`。因为不连带读取庞大的流水数据，响应会达到毫秒级。
3. **扩展性零成本：** 假设下个月您加入了一个极其复杂的机器学习预测策略，参数有 20 个，流水里多了一个 `confidence_score`。您的数据库 DDL 完全不需要变动，直接往 JSON 字段里塞就行。
